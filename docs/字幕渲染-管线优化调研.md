# 字幕渲染管线优化调研报告

> 面向接手开发者的方案调研与路线规划。
> 目标:**在完整保留现有全部特效的前提下**,优化「预览」与「导出渲染」两条管线的性能,
> 并让特效体系（Sayatoo 信号模块、ユートピア 及未来新增特效）可持续扩展。
>
> 约束（硬性，先写在最前面；★ 标注的是 2026-06-23 与产品方确认的口径）：
> 1. **不能砍特效**。Sayatoo 信号灯/音量条、ruby 注音、竖排、glow、双行、分色(roles)、
>    入退场/逐字动画，全部要保留，且未来还会加新的特效族。
>    - ★ **ユートピア(Utopia) = 内部特效，"字体动态跳动"**（逐字随时间做位移/缩放/旋转等仿射抖动；
>      入场放大→收缩、扫光过冲、退场淡出）。**代码里已存在**（`_UTOPIA_*` 常量、entry/exit 的 `"utopia"`、
>      `_paint_line_with_character_transition`）。**★ 该特效可能视觉上"跨行"**（过冲/位移可能侵入相邻行区域），
>      但当前产品决策为**按单行拆分 scope**，见 §5.2。
>      （注：早前文档里的 "コドビーア" 系打字笔误，实为 ユートピア，全文已订正。）
> 2. 跨平台（Windows 优先，但实现选型不要锁死单平台）。
> 3. ★ **性能目标**：
>    - **导出**：在性能允许范围内尽可能快（无固定倍率硬指标，越快越好）。
>    - **预览**：稳定 **24–60 fps**，并**实时显示当前帧率**；**不要求 4K**，**输出分辨率允许用户可调**（低分辨率换流畅）。
> 4. ★ **一致性**：允许导出与预览存在差异，但**差异要尽可能小**；**预览可以不用 GPU**。
>    → 直接结论：**优先保留"单一共享 CPU 渲染器"**（预览与导出走同一套 layout+render，靠同构天然保证一致），
>      **GPU 从主线降级为"可选的导出加速"**（见 §4 方案 C 调整）。
> 5. ★ **句内逐段多段文字风格（字体/字号/颜色混排）的具体视觉规则待定**（§4.5 的对齐策略等留作后续设计）。
> 6. 渐进式：能 A/B 切换、能回退，不要一次性大重写上线。

## 0. TL;DR（结论先行）

- 当前两条管线：**导出**是单线程 Python 逐帧 `QPainter` 软件光栅化 → `constBits` 拷字节 → ffmpeg pipe；
  **预览**已经是较优形态（`QGraphicsView` + 原生 `QGraphicsVideoItem` 视频 + 透明 `QPainter` 字幕层）。
- **真正的逐帧热点**：`painter.py` 已把「未唱层」(阴影+描边+底色) 和「glow」烘焙成位图缓存，
  但**「已唱层」(扫光 fill + stroke + stroke2) 仍每帧用 `addText`+`strokePath` 重新光栅化**，
  只是套了个 clip rect（见 [`_paint_after_path`](../krok_helper/subtitle_render/engine/painter.py)）。这是最大、最易摘的果子。
- 业界共识（libass / AE / SDF 文献）高度一致：**渲染成本由「总位图尺寸 × 每帧需要重新光栅化的量」支配**，
  优化的核心是**「把与时间无关的内容烘焙成位图缓存，逐帧只做位图合成 + clip/transform」**。
- 推荐分三期推进，且**互不冲突、可叠加**：
  - **P0 增量优化**（不改架构，纯收益）：烘焙已唱层、只渲染字幕条带、多进程导出、缓冲复用。导出预计 **5–15×**。
  - **P1 layout/render 解耦 + 图层合成器重构**（中改架构，换可扩展性 + 缓存友好 + 一致性）：
    把渲染拆成 **layout（纯几何）→ animation（逐帧变换）→ paint（位图合成）** 三段，每个特效抽象成「图层/节点」。
    这是让 Sayatoo / ユートピア（字体跳动，视觉跨行但按单行拆分）/ 句内混排 / 未来特效**长期可维护**、且**预览=导出一致**的关键（见 §4.6、§5）。
  - **P2 GPU（可选，且仅导出加速）**：因约束 4「预览可不用 GPU + 要一致」，**不把预览 GPU 化**；
    GPU 仅作为**导出端可选加速**（skia-python(Ganesh) 优先）。**非必需**——CPU 路径缓存做到位通常已够（libass 即无 GPU）。
- libass/ASS 旁路只能覆盖「无 Sayatoo 信号」的普通卡拉OK，**不能作为主路径**（表达不了信号模块），
  但可作为「快速导出」可选项。
- ★ **关于"layout 是否必须与 render 解耦"**（新增确认点）：**P0 不必须**；但要同时满足"预览=导出一致 + 字体跳动 +
  句内混排 + 缓存 + 并行"，**解耦是总枢纽，P1 实质必须**。详见 §4.6。

## 1. 现状盘点

### 1.1 导出管线 `engine/renderer.py`

```
for index in range(total_frames):              # 单线程 Python 循环
    t_ms = round(index * 1000 / fps)
    image = QImage(1920×1080, RGBA8888)         # 每帧新分配 ~8MB
    image.fill(0); paint_frame(image, …)        # QPainter 软件光栅化
    process.stdin.write(bytes(image.constBits)) # 拷 8MB → 写 ffmpeg pipe
```

- ffmpeg 侧：rawvideo rgba pipe 作为 overlay 叠加在缩放后的背景视频上，音频从背景视频拷贝
  （[`build_render_command`](../krok_helper/subtitle_render/engine/renderer.py)）。
- 默认 60fps、1920×1080。60fps 下每秒：60 次全幅光栅化 + 60×8MB≈**470MB/s** 内存拷贝/pipe 带宽。
- 瓶颈：**单核 Python 串行**；`QMediaPlayer` 不参与导出，纯 `QPainter`。

### 1.2 预览管线 `frontend/preview_graphics.py`（默认）/ `preview_view.py`（raster 回退）

- 默认 `PreviewGraphicsView`：`QGraphicsScene` 里放
  - `QGraphicsVideoItem`（视频走 Qt 原生多媒体 → GPU 呈现，**不经过你的 QPainter**）；
  - `SubtitleGraphicsItem`（透明层，`paint()` 调 `paint_frame_to_painter`）。
- 环境变量 `KROK_SUBTITLE_PREVIEW=raster` 回退到老的 `PreviewCanvas`（手画视频帧 + 字幕，更慢）。
- 播放时钟：`TransportBar` 用 `QElapsedTimer` 60/120Hz tick 推进，`QMediaPlayer` 只负责音频
  （Windows 后端 `positionChanged` 粒度粗，所以自己插值）。
- 评价：**视频部分已经不是瓶颈**（原生 GPU）。预览的痛点只剩「字幕层每 tick 全量 `QPainter` 重绘」，
  与导出热点同源。

### 1.3 渲染核心 `engine/painter.py`（~4756 行，特效全在这里）

已有的缓存（做得不错）：

| 缓存 | 内容 | Key（与时间无关） | 失效 |
|---|---|---|---|
| `_BEFORE_LAYER_CACHE` | 未唱层：阴影+描边+底色 烘焙成透明 QImage | line text/font/style/颜色/描边/装饰 | LRU 64 |
| `_AFTER_GLOW_CACHE` | 整行 glow 模糊位图 | 同上 | LRU 64 |
| `_IMAGE_FILL_CACHE` / `_IMAGE_BRUSH_CACHE` | 图片填充 brush | 图片+尺寸 | LRU 16 |

逐帧仍然重算的部分（**热点 + 优化空间**）：

1. **已唱层 fill+stroke+stroke2**：`_paint_line_static` 每帧 `line_path = _line_text_path(...)`
   （`addText` 重建整行轮廓），再 `_paint_after_path` 在 clip band 下 `strokePath`/`fillPath`。
   → **几何逐帧不变，变的只有扫光 clip 边界**。这正是 libass 的 "clip gradient" 技巧适用处，
   而 before-layer/glow 已经这么做了，**唯独 after 主层没做**。
2. **Sayatoo 信号模块** `_paint_signal_lits`：逐帧画若干 `drawRect/Ellipse/RoundedRect`。
   形状简单、量小，但 countdown 状态逐帧变（active_index/phase/flash），属于「天然动态」，
   可优化空间有限（小图集化即可，优先级低）。
3. **入退场/逐字动画** `_paint_line` / `_paint_line_with_character_transition`：opacity/translate/缩放，
   属于「动态变换」，本就便宜（变换是几乎免费的，见 libass 结论）。
4. **每帧重建 `QFont`/`QFontMetrics`/path 对象**：Python 对象分配开销，量大时不可忽略。

### 1.4 分色(roles) 现状

- 当前 `_style_for_line(style, line)` 按 **`line.singer_id`（整行一个歌手）** 覆盖 font_family/
  font_size_px/颜色/描边/glow 等 → **字体/字号可「按行(按歌手)」不同**。
- 句内：`_paint_line_static` 整行只建**一个** `font` + 一个 `metrics`；唯一的句内字体变化是 `font_for`
  的「日文↔英数」family 切换，**且同一字号**。
- 关键：**分色颜色在烘焙位图时就画进像素了，扫光仍只是横向 clip**。所以只要把已唱层也烘焙，
  **分色不会增加任何逐帧成本**，cache key 带上 role 颜色/字体签名即可。这点对 P0 很重要。
- ⚠️ **当前不支持「同一句内逐段不同字体/字号」**（详见 §4.5 专题，这是已确认的新增需求）。

## 2. 瓶颈量化建议（先量再优化）

动手前务必建一个**基准测量脚手架**，否则无法评估收益、也无法防回退：

- 导出侧：固定一段样例字幕（含分色 + 信号 + glow + ruby），统计
  - 纯 `paint_frame` 单帧耗时（不含 ffmpeg）；
  - end-to-end 导出 fps（含 pipe + ffmpeg）；
  - 拆分：光栅化 / `constBits` 拷贝 / pipe 写入 各占比。
- 预览侧：`paintEvent`/`paint()` 单次耗时，播放时实际刷新率、掉帧率。
- 建议做成 `pytest -m bench` 或一个 `scripts/bench_render.py`，输出 CSV，方便 A/B。

经验先验（基于代码结构，待实测校准）：导出端**光栅化**和**8MB/帧 字节搬运**大概率是两座大山；
其中光栅化里「已唱层逐帧重描边」占大头。

## 3. 业界做法调研（可借鉴点）

### 3.1 libass / Aegisub（ASS 卡拉OK 的事实标准）

- Aegisub 只是编辑器；真正渲染器是 **libass**（C 写，mpv/VLC/ffmpeg 内置，跨平台，比 VSFilter 快很多）。
- **铁律一**："渲染性能的支配因素是**总位图尺寸**(total bitmap size)" —— 不是路径复杂度、不是动画数量。
- **铁律二**：多级缓存依赖链 **Font → Outline → Bitmap → Composite**，逐层复用；
  在某个时间戳取帧时，**改颜色/变换/淡入不增加成本**（底层位图被复用）。
- 卡拉OK：每个 shape 拆成最多 4 张**单色 alpha 位图**（`\1c` 填充 / `\3c` 描边 / `\4c` 阴影 /
  `\2c` 卡拉OK高亮），逐帧只做 **alpha 混合 + clip**。分色超过 4 色就拆多 run，但每 run 仍只光栅化一次。
- **"clip gradient" 技巧**：整行渲染一次缓存，再用 N 个不同 clip rect 复用 → 逐帧只是输出不同。
  **= 我们 P0 要补的「已唱层烘焙 + clip」**。
- libass **没有 GPU 后端**（[issue #704 至今 open]），它靠自研快速光栅器 + 激进缓存达到高性能。
  → 启示：**CPU 路径只要缓存做到位，就能很快**，不必为了性能强上 GPU。

### 3.2 After Effects / Premiere（图层合成器架构）

- AE 用 **effect graph + cross-render caching（跨渲染缓存）**：图里没变化的节点保留缓存，
  只有变化的节点重算；"靠近图顶的调整比靠近底部快"。
- 图层 buffer 只在单次渲染 pass 内持有，控制显存/内存占用。
- GPU 加速合成（混合、模糊、变换在 GPU 做）。
- **启示**：把"每个特效 = 一个图层/节点"，框架统一判定「该节点是否随时间变化」→ 决定缓存或重算。
  这正是让 **Sayatoo / ユートピア / 未来特效共存且各自可优化**的工程范式（见 §5 架构建议）。

### 3.3 SDF / MSDF GPU 文字渲染（GPU 路线的关键技术）

- 把字形预生成**(多通道)有符号距离场**纹理；着色器里 **一次纹理采样 + 简单阈值算术**即可得到
  填充 / 描边 / 阴影 / **柔光**，且**任意缩放都清晰**、动态改色零成本。
- msdfgen / msdf-c（基于 stb_truetype 的单文件实现）可离线/运行时生成图集。
- **启示**：若走 GPU，卡拉OK 扫光 = 着色器里按音节移动一个 UV 阈值；分色 = 每 quad 顶点色；
  描边+glow = 距离场算术（省掉多遍描边）。是「重特效 + 4K + 实时」的终极答案，但要重写"观感"。

### 3.4 Skia（skia-python）/ Qt RHI（中间路线）

- **Skia**：跨平台 2D 库（Chrome/Android/Flutter 引擎），有 **GPU 后端(Ganesh, GL/Vulkan/Metal)**、
  自带**字形图集缓存**、可**headless 离屏 GPU 渲染**。`skia-python` 提供 Python 绑定。
  绘制模型与 QPainter 高度同构（`QPainterPath→SkPath`、`strokePath→SkPaint stroke`、
  `QGraphicsBlurEffect→SkMaskFilter blur`），**移植成本远低于裸 GL**。
- **Qt RHI**：Qt 自己的硬件抽象（OpenGL/D3D/Metal/Vulkan）。`QRhiWidget`/`QCanvasPainter` 可做
  GPU 加速 2D。但 PyQt 暴露面有限、生态资料少。
- 注意：**`QPainter` 当前是软件光栅器**（Qt 官方明确："QPainter 目前为软件渲染设计，图形密集场景会成瓶颈"）。
  把 QPainter 画进 `QOpenGLWidget` 也只是合成走 GPU、光栅化仍在 CPU，提升有限。

### 3.5 Python 离屏并行渲染

- `QGuiApplication` + `QT_QPA_PLATFORM=offscreen` 可在**无显示器/worker 进程**里跑 `QImage`+`QPainter`
  （字体需要 QGuiApplication）。→ **多进程导出**可行：N 个 worker 各渲一段帧返回 raw RGBA，
  主进程按序喂 ffmpeg。近线性加速。
- `moderngl.create_standalone_context()` / skia-python 也支持 headless GPU，便于未来 GPU 导出。

## 4. 方案评估

> 评估维度：性能收益 / 改动量 / 风险 / **是否保留全部特效** / 扩展性 / 跨平台。
> 几个方案**不是互斥**的，P0 是所有路线的共同地基。

### 方案 A — P0：增量优化现有 QPainter 管线 ★推荐先做

不改架构，只补缓存与并行。**零特效损失**。

| 子项 | 做法 | 预计收益 | 风险 |
|---|---|---|---|
| **A1 烘焙已唱层** | 把 fill+stroke+stroke2 的「完整一行」烘焙成透明 QImage（与 before-layer 同构的 LRU 缓存，key 含 roles 颜色签名），逐帧把 `_paint_after_path` 改成 `drawImage`+`setClipRect(扫光带)` | 单帧光栅化大降（两次描边+一次填充 → 一次 blit）；**分色零额外成本** | 低，改动集中在一处，有 `test_subtitle_render_painter.py` 兜底 |
| **A2 只渲染字幕条带** | 按 style 布局算出文字 y 范围，只生成/序列化窄条（如 1920×360），ffmpeg `overlay=0:Y` 贴回 | 光栅化/拷贝/pipe 带宽按条带高度线性下降（常见 3×+） | 低-中，需正确处理竖排/双行/信号的纵向范围 |
| **A3 多进程导出** | `QT_QPA_PLATFORM=offscreen` worker 池，各渲一段帧 → 主进程按序写 ffmpeg | 近线性（8 核≈6–7×） | 中，需处理字体一致性、worker 生命周期、取消/进度 |
| **A4 缓冲复用 + 空帧短路** | 复用单块 QImage（`fill(0)`）；无活跃行帧直接写预存全零字节，跳过 paint | 减分配 churn；空段几乎免费 | 低 |
| **A5 对象缓存** | 缓存 `QFont/QFontMetrics`/line path（按 style+text），减少逐帧 Python 分配 | 中等，量大时明显 | 低 |
| **A6 预览脏区** | 预览只 `update(dirtyRect)`（字幕实际区域），降低每 tick 重绘面积 | 预览 CPU 降 | 低 |

- **优点**：风险最低、收益大、不动特效、各子项可独立上线/回退。
- **缺点**：painter.py 仍是 4756 行的巨石，新增特效仍要在里面塞逻辑（扩展性没改善）。
- **跨平台**：✅ 全平台。
- **开发路径**：A1 → A4/A5（同区域顺手）→ A2 → A6 → A3。建议每步配 §2 基准对比。

### 方案 B — P1：图层合成器 + 分层缓存重构 ★推荐作为中期主线

把渲染抽象成「图层栈 + 统一缓存框架」，参考 AE cross-render caching / libass composite cache。

核心模型：

```
Frame = compose([
    Layer.render(t)  for layer in active_layers(t)
])
# 每个 Layer 自述：
#   - static_key(style, content)         # 与时间无关的烘焙 key（None = 不可烘焙）
#   - bake(key) -> Bitmap                # 一次性烘焙（框架缓存）
#   - dynamic(t) -> {clip, transform, opacity}  # 逐帧只产出"怎么贴"，不重光栅化
```

- 现有特效逐个落为 Layer：`TextBeforeLayer` / `TextAfterLayer`(clip 扫光) / `GlowLayer` /
  `RubyLayer` / `SignalLitsLayer`(Sayatoo) / `TitleOverlayLayer` / 未来的 `KodbieaLayer` …
- 框架统一负责：LRU 缓存、失效（style/content hash）、烘焙/合成调度、条带裁剪、并行。
- 新增特效 = 实现一个 Layer 接口，**不用再碰巨石函数**，且自动获得缓存/并行红利。

- **优点**：扩展性质变（Sayatoo/ユートピア/未来特效即插即用）；缓存策略集中、可测；为 P2 GPU 铺平
  （Layer 接口可换 GPU 实现）。
- **缺点**：重构工作量中-大；需把 painter.py 现有逻辑搬迁、保证逐特效像素一致（要做回归对比图测试）。
- **风险**：中。务必**逐特效迁移 + 像素回归**（渲染参考帧 diff），用 env 开关 A/B 与旧路径并存。
- **跨平台**：✅。
- **开发路径**：先定义 Layer 接口与 compositor + 缓存框架 → 迁移 before/after 文本层（已是 P0 形态，最易）
  → glow/ruby → Sayatoo 信号 → title → 收口删旧路径。

### 方案 C — P2：GPU 后端（MSDF 图集 / skia-python） 按需，长期

两条子路线：

- **C1 skia-python(Ganesh)**：用 Skia 替换 QPainter 作为 2D 后端。绘制模型同构，移植成本相对低；
  自带字形图集 + GPU；可 headless GPU 导出。**中间路线**，性价比最高的 GPU 方案。
- **C2 裸 GPU（QRhi/OpenGL + MSDF）**：字形图集 + 着色器，扫光/描边/glow/分色全在 shader。
  天花板最高（4K/实时/超重特效），但要**重写所有特效的观感**，工作量最大。

- **优点**：实时 4K、重特效无压力、导出可 GPU 并行。
- **缺点**：大重写；需重做描边/glow/信号的视觉以达成与现有一致或更好；GPU 驱动/跨平台坑。
- **风险**：高。仅当 P0+P1 后仍不满足（例如要做实时 4K 预览或海量特效叠加）才上。
- **前置条件**：必须先有 P1 的 Layer 接口，GPU 才能作为"另一种 Layer 后端"渐进替换，而非推倒重来。

### 方案 D — libass/ASS 导出旁路 可选加速档，非主路径

普通卡拉OK（无 Sayatoo 信号、无超出 ASS 表达力的特效）时，由 `TimingTrack` 生成 `.ass`
（`\k` 逐音节、`\1c/\2c/\3c` 上色、分色用多 `\c` run、glow≈`\blur`+border、ruby 用定位文本），
交给 `ffmpeg -vf ass=xxx.ass` 一遍硬烧，**没有 Python 逐帧循环**。

- **优点**：导出可快一个数量级；渲染器是全球最经验证的 libass。
- **缺点（致命限制）**：**表达不了 Sayatoo 信号模块、ユートピア 等自定义特效**，无法与现有观感像素一致。
- **定位**：仅作"当前工程不含信号/自定义特效时"的**可选快速导出**，主路径仍是 QPainter/未来 GPU。
- **风险**：低（旁路，不影响主路径）。但要做好"能否走 ASS 旁路"的特性探测与 fallback。

## 4.5 专题 · 句内逐段不同字体/字号（已确认的新增需求）

> 背景问题："现在的缓存方式不允许在一句话里面字体、字号不一样,是真的吗?" —— **半真,但归因要纠正。**

### 4.5.1 事实核实（基于代码）

- **现象属实**：目前一句话**做不到任意逐段不同字体/字号**。
  - `_paint_line_static` 整行只建**一个** `font = _build_font(style)` + 一个 `metrics`
    （[painter.py 约 2009 行]）。
  - 该 `style` 来自 `_style_for_line(style, line)`，按 **`line.singer_id`（整行一个歌手）** 覆盖
    `font_family/font_size_px/...`（[painter.py 约 3939 行]）。→ **字体/字号能"按行(按歌手)"不同,句内不行。**
  - 句内唯一的字体变化是 `font_for` 的「日文↔英数」family 切换（`_make_font_for`），**且同一字号**。
- **但"是缓存不允许"这个归因不准确**：真正的限制在**布局/绘制路径假设了「一行一种字体一个字号」**；
  缓存只是**被动按这个模型建 key**——`_before_layer_cache_key` 里只放了单个 `font_sig` + 单个
  `latin_sig`（[painter.py 约 3464 行]），`_build_before_layer` 也只用单 font 烘焙。

### 4.5.2 关键结论：缓存方案与"句内混排"不冲突，反而是其理想底座

烘焙整行（P0 的 A1）本质是"把**实际字形**画进位图一次"。无论这些字形用了几种字体/字号，烘焙完都只是
**一张位图**，卡拉OK 扫光仍然只是**横向 clip**。所以：

> **"句内逐段混排字体/字号" 与 "管线优化" 是叠加、相互增强的，不是二选一。**
> bake-the-line（A1）天然支持混排；只需把布局改成逐段、并把 per-run 字体签名并入 cache key。

### 4.5.3 要做的工作（与缓存无关，缓存只需扩 key）

1. **布局逐段化**：per-char/per-run 用各自 `QFontMetrics` 取步进（`char_widths` 本就是 list，易改）；
   path 烘焙循环本就逐字，可逐字传不同 `QFont`。
2. **统一行盒与基线**（混排字号的真正难点）：句内字号不同会打破"单 `metrics.height()/ascent()`"的假设。
   需要改成**取各段的 `max(ascent)` / `max(descent)`** 来定：
   - before/after 烘焙位图的高度与基线偏移；
   - 卡拉OK 扫光 clip 的纵向范围（现用 `y - metrics.ascent()` / `metrics.height()`）；
   - `line_rect`、`_resolve_baseline_y`、双行/竖排几何。
   并确定**基线对齐策略**（底对齐 / 居中 / 视觉中线），作为风格项暴露。
3. **缓存 key 加 per-run 字体签名**：把单 `font_sig` 换成"逐段 `(family, size, weight, italic)` 的元组签名"，
   不同混排自然落不同缓存项；其余缓存逻辑（LRU、失效、blit+clip）不动。
4. **关联模块同步**：ruby 注音、信号模块的纵向锚点依赖行盒高度，需跟随统一行盒重算。

### 4.5.4 与路线图的关系

- 这本质是 roles 的 **Phase 3（逐段渲染）** 再叠加"字号也可变"。
- **建议在 P1（图层合成器重构）一并落地**：P1 把"行布局"抽成独立步骤后，逐段化与统一行盒是自然的扩展点，
  且 Layer 接口的 `static_key` 直接纳入 per-run 字体签名。
- 若要在 P0 阶段先行支持，也可行（A1 的烘焙天然兼容），但 §4.5.3 第 2 点的"统一行盒/基线"改动会散落在
  多处旧函数里，回归面较大——故**优先放 P1**。
- **验收**：在像素回归测试集中**新增"句内混排字体+字号"参考帧**（含分色、ruby、双行组合）。

### 4.5.5 评估小结

| 维度 | 说明 |
|---|---|
| 现状 | 句内仅 jp/latin family 切换且同字号；字体/字号只能整行(按歌手)变 |
| 根因 | 布局假设单字体单字号；缓存只是跟随，非根因 |
| 与优化的关系 | **正交且互促**——bake-the-line 是混排的理想底座 |
| 主要难点 | 句内混字号下的**统一行盒 + 基线对齐**（不是缓存） |
| 建议落点 | P1 图层合成器重构内一并实现（roles Phase 3 的超集） |
| 风险 | 中（纵向几何改动面广，靠像素回归兜底） |

## 4.6 专题 · layout 与 render 是否必须解耦（新增确认点）

> 问题："layout 和 render 是不是必须要解耦?现在项目内似乎是一个 layout 直接画。"

### 4.6.1 现状核实

属实，**当前 layout 与 render 是耦合的**：`_paint_line_static` 在同一个函数里**既算布局**
（`char_widths`、`x0`、`baseline`、`char_x_ranges`、`intervals`、`fill_segments`）**又直接绘制**
（建 path、画 before/after）。只有少数布局（如 `_resolve_sayatoo_line_layouts` 的信号 union）被单独抽出。
即"一次布局直接画"。

### 4.6.2 是否"必须"解耦？——分阶段回答

- **P0（增量优化）：不必须。** A1 烘焙已唱层、A2 条带、A4 缓冲复用都能在现有耦合结构里就地做。
- **P1 及以后：实质必须。** 你这次确认的四个诉求**全部**指向"layout 必须独立于 render"：

| 诉求（你的澄清） | 为什么需要解耦 |
|---|---|
| **预览=导出一致**（约束 4） | 预览与导出调**同一个 layout**（纯几何，确定性），只有最后 paint 的画布不同 → **一致性由构造保证**，而不是靠两边各写一遍对齐 |
| **ユートピア 字体跳动 + 视觉跨行**（约束 1★） | 跳动 = 逐帧给字形一个**仿射变换**(位移/缩放/旋转)。layout 先算"静止盒子"，animation 段叠加变换，paint 段 blit 已烘焙字形 + 变换 → **不必逐帧重光栅化**（现状是每帧重光栅化，见 §5.2）。视觉跨行按单行 group 的保守 bounds 覆盖，不做跨行联合 scope（见 §5.2.4） |
| **句内混排字体/字号**（§4.5） | layout 段统一算 per-run 盒子 + max(ascent/descent) 行盒；render 段无需关心混排 |
| **缓存 + 多进程并行** | layout（几何）与 bake（位图）都可按"静态签名"缓存；导出时一行只 layout 一次，各帧只跑 animation+paint，天然适合并行 |

> 结论：**P0 可以先不解耦拿快钱；但要达成你要的最终形态，P1 必须把 layout 抽成独立的纯函数阶段。**
> 这也正是 libass（shaping/layout → bitmap → composite 三层）和 AE（effect graph）的通用结构。

### 4.6.3 建议的三段式（layout → animation → paint）

```
# 1) layout：纯几何，无 QPainter，确定性 → 可缓存、预览/导出共用、可并行
FrameLayout = layout(track, style, t_window)
#   每行: [GlyphBox(x, y, font, advance, run_id), ...]，line_box(max_ascent/descent)，
#         fill_segments，ruby_boxes，signal_geometry …

# 2) animation：给定 layout + t，产出"逐帧怎么变"，仍是纯数据（无光栅化）
FrameAnim = animate(FrameLayout, t_ms)
#   per-glyph transform(跳动/入退场)、opacity、karaoke 扫光 clip、信号 active 状态 …

# 3) paint：消费 layout + anim，blit 已烘焙位图 + 套 transform/clip（后端可换）
paint(device, FrameLayout, FrameAnim)   # device = 预览 widget / 导出 QImage / (未来)Skia
```

- **一致性**：预览与导出 1、2 段**完全相同**，只有第 3 段的 device 不同 → 差异天然最小。
- **后端可换**：第 3 段抽象成接口后，未来要给"导出端"接 Skia/GPU，只换 paint 实现，layout/anim 不动，
  **观感不漂移**（直接满足约束 4）。
- **预览不用 GPU**：第 3 段在预览用现有 QPainter 即可；性能靠 layout/bake 缓存而非 GPU。

### 4.6.4 与既有代码的迁移关系

- 把 `_paint_line_static` 拆成 `layout_line()`（返回 GlyphBox/行盒/fill_segments）+ `paint_line()`（消费）。
- `_resolve_sayatoo_line_layouts`、`_signal_lit_groups` 已是"半 layout"，并入 layout 段。
- before/after/glow 的 bake 函数保持不变，只是改由 layout 结果驱动。
- 风险：纵向几何（baseline/行盒/clip 带）改动面广 → 靠 §7 像素回归兜底，env 开关与旧路径并存。

## 5. 针对「特效可扩展性」的架构建议（回应核心诉求）

### 5.1 Effect/Layer 插件接口（建立在 §4.6 三段式之上）

你明确说"还要 Sayatoo + ユートピア + 未来更多特效"。所以**比单点提速更重要的是把特效做成可插拔**。
建议在 P1 落地一个 **Effect/Layer 插件接口**，落在 §4.6 的 `layout → animation → paint` 三段上，
并把"静态可烘焙 / 逐帧动态"二分作为一等公民：

```python
class SubtitleLayer(Protocol):
    def active_window(self, ctx) -> list[tuple[int,int]]: ...    # 该层何时出现（决定可见性/裁剪）
    def layout(self, ctx) -> LayoutData: ...                     # 纯几何（无 QPainter）：盒子/行盒/分段
    def static_key(self, ctx) -> Hashable | None: ...            # 与 t 无关的烘焙 key（含 per-run 字体签名）；None=纯动态
    def bake(self, ctx, key) -> BakedBitmap: ...                 # 一次性烘焙（框架缓存+LRU）
    def animate(self, ctx, layout, t_ms) -> AnimState: ...       # 逐帧只产 clip/transform/opacity（无光栅化）
    def vertical_bounds(self, ctx) -> tuple[int,int]: ...        # 供条带裁剪(A2)与脏区(A6)
```

- 卡拉OK 文本 = `bake(整行) + animate(扫光 clip)`；
- Sayatoo 信号 = 多个 `bake(单个灯/列)` + `animate(active_index/phase 选哪些 + 闪烁 alpha)`；
- **ユートピア（字体跳动，视觉跨行但按单行拆分）= `bake(字形一次)` + `animate(逐字仿射变换)`**，作用域为单行内 group（见 §5.2）；
- 句内混排 = `layout` 段算 per-run 盒子，`bake` 把混排字形画进一张位图，`animate` 仍只做扫光 clip；
- 未来特效 = 各自实现接口，**自动获得缓存、条带裁剪、并行、一致性、(未来)GPU 后端**。
- 框架集中管：缓存与失效、合成顺序(z)、条带范围聚合、worker 调度。

这样"加特效"从"改 4756 行巨石"变成"加一个 Layer 类 + 测一张参考帧",这是长期可维护性的根本。

### 5.2 专题 · ユートピア(Utopia)「字体动态跳动 + 跨行」如何融入 bake 缓存模型

> 已是现有特效。现状：`_paint_line_with_character_transition` 对 `transition.effect=="utopia"` 用
> `_transition_char_state` 产出逐字 **仿射变换**(opacity/dx/dy/rotation/scale_x/scale_y/skew_y)，
> 然后**每帧 `addText` 重建 path + `QTransform.map` + 重绘**（见 painter.py 约 2379–2409 行）——即**每帧重光栅化**。
> 这正是 §5.2.1 要改成"bake 一次 + 逐帧变换"的典型例子。

#### 5.2.1 把"逐帧变化"分四类（绝大多数落 a/b/c，只有 d 才需逐帧重光栅化）

| 类别 | 逐帧变化 | 处理 | 例子 |
|---|---|---|---|
| (a) **裁剪** | 只是显示范围推进 | bake 一次 + 逐帧 `setClipRect` | 卡拉OK 扫光 |
| (b) **变换/透明** | 位移/旋转/缩放/淡入淡出 | bake 一次 + 逐帧 `QTransform`/`opacity` | 入退场动画、**字体位移/旋转跳动** |
| (c) **离散状态选择** | 在少数预烘焙状态间切换 | bake 一组 + 逐帧选 index | 信号灯亮/灭 |
| (d) **连续重光栅化** | 字形轮廓本身连续改变 | 见下方缓解策略 | **字号连续脉动**、字重 morph |

ユートピア 现用的是 **位移+旋转+skew（属 b）+ 入场/扫光的缩放过冲（1.3×/1.15×，属 b 含缩放）**：

- **位移/旋转/skew** → 纯 (b)：字形 bake 一次，`animate` 每帧给该字一个 `QTransform`，
  paint 段 `drawImage` 带变换，**逐帧成本≈一次 blit**。
- **缩放过冲（放大→收缩）** → 走"**超采样 bake + 逐帧缩放**"：bake 在**过冲后的最大字号**（如 1.3×），
  逐帧缩小（清晰、放大不糊）。这样把现状的"每帧 addText 重光栅化"降为"一次 blit + transform"。
- 若未来出现"字号/字重**连续脉动**"这种真 (d)，缓解：(1) 量化成 N 档字号各 bake 一份（退化成 c，逐帧选档）；
  (2) 超采样 bake 缩放；(3) 只对"正在跳动的少数字"逐帧重 bake（跳动通常只作用于当前唱到的字，量很小）。

> 设计要点：跳动是 **animation 段**的职责（产出 per-glyph transform/选档），**不污染 layout 段的静止盒子**；
> 这样跳动与"句内混排/缓存/并行/一致性"全部解耦。**这也是为什么 §4.6 的解耦对 ユートピア 是刚需。**

#### 5.2.2 现状 vs 目标

| | 现状 | P1.c 目标 |
|---|---|---|
| 逐帧 | `addText` 重建 path + `transform.map` + 重绘 | bake 一次 + 逐帧 `QTransform` blit |
| 缩放过冲 | 每帧重光栅化 | 超采样 bake（最大字号）+ 逐帧缩放 |
| 作用域 | 单行内的字符组（ruby group） | **单行内 group；视觉跨行只扩 bounds**（见 §5.2.4） |

#### 5.2.3 关于精度/观感

bake+变换 与 每帧重光栅化 会有**极小的边缘抗锯齿差异**（放大时尤其）。用超采样 bake（按最大过冲尺寸）可把差异
压到肉眼不可见。务必纳入 §7 像素回归（utopia 入场/扫光/退场各取若干帧对比）。

#### 5.2.4 ★ "跨行"对架构的影响（重点）

**2026-06-23 产品决策更新**：如果一个短语因为排版视觉上跨到两行，当前不把它作为一个跨行动画单元合并处理；
按**单行拆分**。也就是说，utopia 的 `group` scope 只在单行内聚合 ruby/主字组，`scope_id`
必须包含行边界（如 `line_start` / `line_end`），防止两行里相同短语被合成一个跨行 group。
放大过冲侵入相邻行只作为该单行 group 的保守 bounds 覆盖，不驱动跨行动画同步。

早期方案曾按"跨行联合 group"设计；当前决策改为**视觉跨行但按单行拆分**。仍然需要在合成器里引入
**"特效作用域(scope)"** 概念，但 utopia 的 `group` 只聚合单行内的 ruby/主字组：

- **scope 级别**：`glyph`（单字）/ `line`（单行）/ `group`（单行内短语/熟字训/ruby 组）/ `frame`（全屏）。
  ユートピア 的动画单元是单行内 `group`，普通卡拉OK 扫光是 `line`，标题 overlay 是 `frame`。
- **layout 段**：产出单行内 group 的联合盒子；`scope_id` 必须包含行边界，避免相邻行相同短语被误合并。
- **bake/clip**：不做跨行联合裁剪；放大过冲或位移侵入相邻行时，用当前行 group 的保守 bounds 覆盖。
- **active_window/可见性**：按行内 group 判定；短语折到两行时拆成两个 group，各自跟随所在行。
- **条带渲染(A2) 与脏区(A6)**：纵向范围聚合当前行 group 的保守 bounds；不再计算跨行 group 联合高度。

> 这进一步印证 §4.6：**layout 必须独立于 render，且要有明确作用域**。Layer 接口的
> `active_window/layout/vertical_bounds` 都带 `scope` / `scope_id`，框架按 scope 聚合几何与裁剪范围。

### 5.3 待你确认（ユートピア 行为细节）

1. 缩放过冲的最大倍率是否就是现有常量（入场 `1.3×`、扫光 `1.15×`）？决定超采样 bake 的目标尺寸。
2. "跨行"具体形态：短语折到两行时按单行拆分；上一行过冲视觉侵入相邻行时，只扩当前行 group 的保守 bounds。
当前假设：跳动以 (b) 仿射 + 超采样 bake 处理；视觉跨行按单行 group 拆分，
不做跨行联合盒子。

### 5.2.6 实测与决策：ユートピア bake 化（P1.c-1）

**实测（1080p、glow+stroke 的 utopia 行、入场+扫光阶段帧）**：单帧 ≈ **4.0 ms**。
逐项验证发现：

- 先尝试的"安全优化"——缓存 `_glyph_run_path` 的字形轮廓（避免每帧 `addText` 重取轮廓）——
  **无可测收益**（4.005 → 3.971 ms，噪声内）。说明 utopia 的逐帧成本由**光栅化**
  （`strokePath` / `fillPath` / glow 模糊）支配，而非轮廓提取。该缓存已回退（不留死代码）。
- 因此 utopia 唯一真实的提速是 **bake 一次 + 逐帧变换**（把每帧的两次描边+填充+glow 光栅化
  降为一次 blit+QTransform，预计 ~4×）。但 bake 出的位图经 `QTransform` 缩放会引入**边缘抗锯齿
  软化**（放大相位尤甚）——即**改变 flagship 特效的像素**。

**张力**：这与产品方既定约束「**保留当前特效功能** + 导出/预览**差异尽量小**」直接冲突——
bake 化是 perf 与 视觉保真 的取舍，**不是纯透明优化**。故 P1.c-1 **挂起待确认**，不擅自改 flagship 观感。

**若确认要做，推荐实现路径（可逐步、每步像素/行为回归）**：
1. **超采样 bake**：按固定 `S=2×`（覆盖 1.3× 入场过冲且始终降采样→锐利）烘焙 group 的
   before/after/(glow) 位图。新增 `_build_glyph_run_layer` 的 `supersample` 变体（字号×S、
   坐标×S、画进 S² 面积位图）。
2. **逐帧合成**：`painter.save(); setClipRect(屏幕系扫光带)` →（Qt clip 在设时即落 device 空间，
   后续 transform 不动它）→ `setTransform(group_transform ∘ scale(1/S), combine=True)` →
   `drawImage(before/after/glow)`。before 全画、after+glow 按扫光带 clip——复刻
   `_paint_char_karaoke_stack` 的 before→glow→after 顺序与裁剪。
3. **回归**：utopia 入场/扫光/退场各取帧，behavior 哈希断言保持；新增 bake-vs-vector 的
   容差像素对比（SSIM / 允许小阈值），并请产品方目视签收一次"软化是否可接受"。
4. **作用域**：复用 §5.2.4 已落定的"单行 group"（不跨行）。倍率沿用现有常量（§5.3 第 1 项）。

**已实现范围（opt-in 当前版本）**：覆盖 utopia **主文本**（role 与 plain 两条 transition 路径）。utopia
**ruby** 仍走逐帧 vector（小、glyph 少，perf 可忽略）；开关开启时主文本为 bake（边缘略软）、ruby 为 crisp
vector，存在极轻微观感不一致——属已知的次要 follow-up，待确有需要时再把 ruby transition 也 bake 化。

## 5.4 专题 · GPU 与 CPU 渲染的差异（你要的：到底差在哪，写进文档）

> 你问"GPU 和 CPU 预览/渲染/导出的差异是什么"。下面分**(A) 性能差异**和**(B) 观感/正确性差异**两面讲清，
> 并给出**为什么本项目当前结论是"预览与导出统一用 CPU(QPainter)，GPU 只作可选导出后端"**。

### 5.4.1 名词先对齐（本项目语境）

- **CPU 渲染** = 现状：`QPainter` **软件光栅器**把矢量/字形扫描转换成像素（Qt 官方明确 QPainter 为软件渲染设计）。
  预览的字幕层、导出的每帧都走它（预览的**视频**另由 `QGraphicsVideoItem` 走原生 GPU，但那只是"播放视频"，
  不是"渲染字幕"）。
- **GPU 渲染** = 候选：把字形做成**纹理图集**，用着色器画带纹理的四边形（OpenGL/Vulkan/Metal/D3D，
  或经 Skia Ganesh / Qt RHI 抽象）。

### 5.4.2 (A) 性能差异

| 维度 | CPU(QPainter 软件光栅) | GPU(图集 + 着色器) |
|---|---|---|
| 单帧光栅化 | 受字形数/描边/blur 影响，单核 | 并行像素填充，量大时快很多 |
| 扩展到 4K / 海量特效 | 吃力 | 优势明显 |
| **本项目实际瓶颈** | 主要是"**每帧重光栅化**"，靠 bake 缓存即可大幅缓解（见 P0/P1） | — |
| 并行导出 | 多进程即可近线性（约束已满足） | 需 headless GPU 上下文，复杂 |
| 结论 | **缓存做到位通常已够**（libass 无 GPU 也快） | 仅当 CPU 路径仍不够时才需要 |

> 关键认知：本项目的慢**不是"CPU 比 GPU 慢"那么简单**，而是"该缓存的没缓存、单核串行"。先把 P0/P1 做了，
> CPU 路径大概率就满足你的目标（导出尽快、预览 24–60fps）。GPU 是"还不够时的下一档"，不是必选。

### 5.4.3 (B) 观感 / 正确性差异（**这是要统一渲染器的根本原因**）

同一套矢量+样式，CPU 与 GPU 画出来**不会逐像素相同**，常见差异：

| 差异点 | 说明 | 后果 |
|---|---|---|
| **抗锯齿算法** | CPU 用高质量覆盖率扫描转换 AA；GPU 常用 MSAA / 着色器覆盖近似 | 文字/描边**边缘软硬不同**，细字尤其明显 |
| **字形光栅化与 hinting** | CPU 经 FreeType/系统字体光栅器（带 hinting、子像素定位）；GPU 图集多为"某尺寸 bake 一次再缩放" | 字形**粗细/清晰度/对齐**不同，缩放时 GPU 略糊 |
| **渐变插值色彩空间** | sRGB 还是 linear 空间插值，两端实现常不一致 | **渐变中段颜色/条带**不同（你的 fill 渐变、split 渐变会受影响） |
| **模糊/glow** | CPU `QGraphicsBlurEffect`（高斯）vs GPU 着色器 blur，半径语义/衰减不同 | **glow 大小与柔和度**不同 |
| **混合与 premultiplied alpha 舍入** | 半透明叠加的取整方式不同 | 多层叠加后**累积 1–2 级色差**，半透明边缘可见 |
| **裁剪边缘 AA** | 矩形 clip：CPU 多为硬边，GPU 可能 MSAA 软边 | 扫光交界、条带边缘**锯齿/羽化**不同 |
| **浮点精度** | GPU fp32/fp16 vs CPU double | 亚像素**定位舍入**差异，动画时轻微抖动 |
| **跨厂商不确定性** | 同一 GPU 代码在 NVIDIA/AMD/Intel/驱动版本上结果可能不同 | **导出在不同机器上不可复现**；预览(本机 GPU) vs 导出(另一路径)更难一致 |

> 一句话：**CPU 渲染是确定性的、跨机一致的；GPU 渲染受后端/驱动影响、且与 CPU 有系统性观感差异。**

### 5.4.4 对"预览 vs 导出一致性"的直接结论（呼应约束 4）

- **若预览用 GPU、导出用 CPU（或反之）** → 必然出现 §5.4.3 的系统性差异，且 GPU 端还跨机不稳定 →
  违背"预览=导出尽量一致"。
- **若预览与导出都用同一套 CPU(QPainter) 渲染** → **一致性由构造保证**（同一代码、同一光栅器），
  这也正是你说"预览可不用 GPU"所允许的最优解。
- **若将来一定要 GPU 加速导出** → 在 §4.6 三段式下**只替换 paint 后端**，但要付出"重新对齐观感 + 像素回归"的成本；
  且**预览仍保持 CPU**，意味着此时预览与导出会有 §5.4.3 的小差异 → 需产品方接受。
- 折中（若 P2 走 Skia）：让**预览与导出都改用 Skia**（Skia 可 CPU 也可 GPU），至少"同一渲染库"能把差异收窄；
  但仍受 CPU/GPU 后端差异影响，不如纯 CPU 共享来得稳。

### 5.4.5 给接手者的执行性建议

1. **默认**：预览 + 导出共用 CPU QPainter 渲染器（三段式 layout/anim 完全共享，paint 都用 QPainter）。
2. GPU 仅在"P0+P1 后导出仍不够快"时，作为**导出专用可选后端**引入，并：
   - 固定一个参考后端做"基准观感"，其余后端对齐到它；
   - 把 §5.4.3 各项纳入像素回归阈值；
   - 文档标注"GPU 导出与 CPU 预览存在已知小差异"。
3. 预览侧若要更快，优先"**降输出分辨率**（约束 3★，已预留）+ bake 缓存 + 脏区"，而不是上 GPU。

## 7. 风险控制与验收

- **A/B 与回退**：沿用现有 `KROK_SUBTITLE_PREVIEW=raster` 的范式，新路径都加 env/配置开关与旧路径并存。
- **像素回归测试**：对一组覆盖全特效（分色+信号+glow+ruby+竖排+双行+动画）的参考帧，做
  渲染结果 diff（允许极小阈值）。**这是 P1 重构不翻车的生命线**。
- **性能门禁**：把 §2 基准纳入 CI（或本地 `bench`），每个优化前后出对比数,防止"改完更慢"。
- **多进程坑**：字体加载一致性（worker 各自 `QGuiApplication`）、取消/进度回传、异常传播、有序写盘。
- **GPU 坑（若到 P2）**：跨平台驱动差异、headless 上下文、与 ffmpeg 的零拷贝/回读策略。
