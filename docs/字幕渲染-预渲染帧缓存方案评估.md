# 字幕渲染 · 预渲染 + 帧缓存方案评估

> 面向接手者的**单点方案评估**。聚焦一个问题：预览端能否抄 After Effects / Premiere / Nicokara 的
> 「预渲染 + 帧缓存」范式，解决 **utopia（逐帧抖动重特效）实时预览掉帧**。
>
> 配套阅读：主调研报告 [`字幕渲染-管线优化调研.md`](字幕渲染-管线优化调研.md)
> （本方案 = 该文档 §10.5 Tier 2 / §10.7 S2' 的展开评估）。
>
> 触发：2026-06-24 讨论确认 utopia body 单帧已撞 `strokePath` 的 Qt C++ 地板（持 GIL），
> 单线程内无剩余安全提速；业界对「红区重特效」的解法不是「单帧更快」，而是**改架构去预渲 + 缓存**。

---

## 0. TL;DR（结论先行）

- **范式认定**：AE/PR/卡拉OK 软件流畅的根本，是**把「播放帧率」与「渲染帧率」解耦**——不在播放瞬间渲，
  而是提前/空闲渲好、**存成帧缓存**，播放时只读帧。AE 的 RAM Preview、PR 的绿色渲染条、Nicokara 的
  时间线已知预渲，都是这一招。
- **本项目适配度高**：①字幕时间线**全程已知**（无交互不确定性，比通用视频编辑器更适合预渲）；
  ②要缓存的只是**字幕透明层**（视频仍走原生呈现，不进缓存）；③现有 `AsyncSubtitleRenderer`
  已经是「worker 渲进 QImage、GUI blit」的半成品，**改造成本集中、风险可控**。
- **但有一个诚实的前提（决定成败）**：帧缓存解决「**重播 / seek / 抖动突发**」立竿见影；但要解决
  「**首次连续播放 utopia 段**」的流畅，取决于**填充吞吐能否 ≥ 播放速率**——
  - utopia 单线程填充 < 实时 → 单纯 look-ahead **缓冲会被抽干**，不能凭空变快（呼应主文档 §10.3
    「刷新率与同步是两条独立的轴」）；
  - 故要真正顶住首播，必须**二选一**：(A) **AE 式**——先预渲该段（阻塞 + 进度条）再满帧回放；
    或 (B) **多进程填充**（S2'）把填充吞吐抬过实时，边播边追。
- **推荐分档**：
  - **P-A 帧缓存环（小，1 周内）**：把 latest-wins worker 升级为「按帧索引缓存 + 命中即取」。
    立即收益：重播/seek/逐帧 scrub 命中即满帧；为后续两档铺底座。**低风险、先做。**
  - **P-B look-ahead 填充（中）**：空闲时按播放头向前预填环形缓冲。对**含轻松段**的混合工程有效；
    纯重 utopia 段受单线程吞吐限制。
  - **P-C 二选一顶住首播**：(A) AE 式预渲工作区（简单、必满帧、需等待）/ (B) 多进程填充（绕 GIL、
    可边播边追、工作量 L–XL、IPC 风险）。**据产品对「等待 vs 即播」的取舍定。**

## 1. 为什么是这条路（业界范式锚定）

主文档 §10 已确认：当前预览是 **render-just-in-time**——墙钟/音频时钟拽到某时刻，才临时渲那一帧字幕。
这正是 Premiere 时间线的「**黄/红条**」模式：

| PR 渲染条 | 含义 | 本项目对应 |
|---|---|---|
| 🟢 绿 | 该段有**预渲染 preview 文件**，回放读文件 → **保证实时** | **本项目没有**（无帧缓存）|
| 🟡 黄 | 无预渲文件，到播放头临时渲，"可能"实时 | 普通行（有 bake 缓存，够便宜）|
| 🔴 红 | 无预渲文件，临时渲，特效太重"大概率非实时" | **utopia（+glow）= 红区，掉帧**|

- **After Effects** 更极端：没有实时播放，按空格 = 先把工作区每帧渲进 RAM/磁盘缓存（RAM Preview），
  **渲完才按 fps 回放缓存**。官方明言重特效"无法实时同时处理并显示，必须先算好存起来"。
- **共同本质**：**时间换流畅**（提前/空闲算）+ **空间换流畅**（RAM/disk 存帧）+ **多核**（AE 的
  Multi-Frame Rendering 多核并行渲多帧，压短"提前渲"的墙钟）。
- **卡拉OK 特别适合**：歌词时间线播放前**完全已知**，可在播放前/空闲整首预渲缓存，播放纯读帧。

> 结论：utopia 这种红区特效，靠"优化单帧"逼近 C++ 地板、收益有限；**正解是抄 AE/PR——预渲 + 帧缓存 + 多核**。
> 代价是放弃"零延迟实时贴"，引入预渲队列与缓存内存——而卡拉OK 时间线已知，这个代价比通用编辑器更小。

## 2. 当前架构现状核实（grounded）

**预览字幕渲染已是「worker 渲 QImage → GUI blit」**（[`preview_async.py`](../krok_helper/subtitle_render/frontend/preview_async.py)）：

- `AsyncSubtitleRenderer` 持一个 `QThread` worker；GUI 经 `set_state(track,style)` /
  `set_render_target(w,h,dpr)` / `request(t_ms)` 投递。
- worker 是 **latest-wins**：`request` 只存 `self._pending_t = t_ms`；正在渲就丢弃中间请求，渲完取最新
  （[`preview_async.py:80`](../krok_helper/subtitle_render/frontend/preview_async.py#L80)、`_render_pending` :91）。
- 渲完从 worker 线程 emit `frame_ready(QImage, t_ms)` → GUI 的
  [`_on_async_frame`](../krok_helper/subtitle_render/frontend/preview_graphics.py#L272) → `set_async_image(image)` blit。
- 播放驱动：墙钟/音频时钟 tick → `set_time(t_ms)` →
  [`preview_graphics.py:258`](../krok_helper/subtitle_render/frontend/preview_graphics.py#L258) → `request(t_ms)`。

**关键观察**（决定本方案可行性）：

1. **缓存的只是字幕透明层**——视频走原生 `QGraphicsVideoItem`，不进缓存。缓存对象天然小（且大面积透明）。
2. **latest-wins = 主动丢帧**：worker 慢于 tick 时，中间帧被丢 → 字幕刷新率掉到 worker 产能（实测 utopia+glow ~34/s）。
   **这正是没有帧缓存的直接后果**：渲过的帧用完即弃，重播/seek 还要重渲。
3. **worker 是单线程**：`QThread` 内 `paint_frame_to_painter`，GIL 下无法多核（主文档 §10.6 探针已证多线程无效）。
4. 失效信号已存在：`set_state` / `set_render_target` 变更即应清缓存（对齐 `LayerCache.clear` 范式）。

## 3. 方案设计

### 3.1 核心数据结构：按帧索引的帧缓存

```python
frame_index = round(t_ms * fps / 1000)        # 量子化到预览帧率，避免浮点 t 抖动
key = (frame_index, render_target_sig)        # render_target_sig = (w, h, dpr, style_hash, track_hash)
cache: dict[int, QImage]                        # 环形 / LRU，按 frame_index
```

- **缓存值 = 字幕透明层 QImage**（设备分辨率，与现 worker 产出同构）。
- **失效**：`set_state`（track/style 变）/ `set_render_target`（尺寸/DPR 变）→ **整清**（与 `LayerCache` 同范式）。
- **内存预算**（务必先估）：device ~2400×1350 ARGB32 ≈ **13MB/帧**。
  - 1 秒 look-ahead @60fps = 60 帧 ≈ **780MB**——**不可全曲缓存**，必须**环形有界**（如播放头 ±1–2s）。
  - 降本选项：①只缓存**字幕条带**（纵向并集，复用导出 `KROK_SUBTITLE_RENDER_STRIP` 的 band 逻辑）→
    高度常压到 1/3；②透明层 RLE/PNG 压缩（省内存、费 CPU，权衡）；③缓存按 logical 分辨率（牺牲 DPR 锐利）。

### 3.2 三段式递进（每段可独立上线 / A-B 回退）

#### P-A · 帧缓存环（latest-wins → 命中即取） ⛔ 已实现并回退（实测不适用 utopia，2026-06-25）

最小改动、立即收益、零架构风险：

- GUI 收到 `request(t_ms)`：先查 `cache[frame_index]`，**命中 → 直接 blit、不投 worker**；未命中才投 worker。
- worker `frame_ready` → **写入 cache** 再 blit（而非用完即弃）。
- 环形淘汰：保留播放头附近窗口，seek 远离即逐出。
- **收益**：重播同段、seek 回已渲处、暂停后逐帧 scrub → **命中即满帧**（这部分现在每次都重渲）。
  对"首次连续播放"无收益（仍是 latest-wins 填充），但**它是 P-B/P-C 的底座**。
- **风险**：低。env 开关 `KROK_SUBTITLE_PREVIEW_FRAME_CACHE`（默认可先关，A/B 验证后转默认开）。

> **⛔ 实测与回退（2026-06-25）**：曾按上述实现（帧网格量子化 `preview_frame_index` /
> `preview_frame_canonical_ms`、`AsyncSubtitleRenderer` 内置有界 LRU、token 水位丢过期帧、
> `set_state`/`set_render_target` 失效，含 12 例单测）。**真机开 `KROK_SUBTITLE_PREVIEW_FRAME_CACHE=1`
> 在 utopia+ruby 工程上目检失败**，已整体回退（`preview_async.py` 恢复到无缓存形态、删除单测）。原因——
>
> 1. **对核心痛点（utopia 前向播放掉帧）零收益**：播放时钟固定 **60Hz tick**（`preview_view.py`
>    `_DEFAULT_PREVIEW_FPS=60`，注释明载历史上从 120Hz 改回、多余 tick 会被合并丢弃）。60Hz tick + 60fps
>    缓存网格 → 每 tick 落不同帧桶 → **首次连续播放全 miss、每帧照样重渲**。P-A 设计上只加速重播/seek/scrub
>    （§4 表已写明），而用户要解决的恰是首播 utopia 掉帧 → **用错了工具**。
> 2. **会破坏 utopia 观感（致命）**：tick 抖动偶尔使两 tick 落进同一帧桶 → 第二 tick **命中**返回该桶
>    **起点时刻**的 utopia 帧；utopia 每帧都在抖动/缩放（扫光过冲 1.15×），正被扫光的字就比应有位置
>    **差最多 16ms** → 该字明显错位/重影（用户截图「移」字的 bug）。**这与 §9.7 B2 否决量子化重栅是同一根因**：
>    utopia 动画比量子步长快得多，任何「缓存帧/吸附时间再复用」对正在动画的字都会时间不一致。
>
> **教训**：帧缓存只对**逐帧位图相同**的内容（静止保持段、非动画）有效；对 utopia 这类「每帧都不同」的
> 旗舰特效**根本不适用**——与 §8.3 的「相同帧短路」结论一致（utopia 无效）。**utopia 前向播放掉帧的正解
> 不是缓存，而是真并行（多进程 S2'，§10.7）或 look-ahead 配多进程把填充吞吐抬过实时。**
>
> 若未来仍要做帧缓存，须**仅对静止帧生效**（整句唱完保持 / 句间停顿；utopia/扫光/入退场进行中的帧绝不缓存），
> 但对以 utopia 为主的工程收益很小，性价比低（见 §3.2 P-A「窄化修复」讨论）。

#### P-B · look-ahead 预填（空闲向前渲）

- worker 空闲（无 pending 当前帧）时，按播放方向预填 `frame_index+1, +2, …` 到环形上界。
- 播放推进时优先吃预填好的帧。**对含轻松段的混合工程**：轻松段 worker 跑得快、攒出缓冲，
  到重 utopia 段消化缓冲 → 平滑掉帧突发。
- **诚实边界**：若**全程重 utopia**（无轻松段可攒），单线程平均填充率 < 实时 → **缓冲稳态被抽干**，
  退化回 P-A 行为。**不能凭空把单线程产能变高**（主文档 §10.3）。
- **风险**：中。需处理 seek 时丢弃错误方向的预填、暂停/播放切换、预填不抢占当前帧请求。

#### P-C · 顶住"首次连续播放重 utopia"（二选一，据产品取舍）

**(A) AE 式 · 预渲工作区后回放** —— 简单、必满帧、需等待：

- 用户对重段先触发"预渲"（或自动检测红区）→ 阻塞渲该区间进缓存（带进度条）→ 完成后满帧回放。
- 工作量 **S–M**；纯复用 P-A 缓存 + 一个顺序填充循环 + 进度 UI。
- 取舍：牺牲"即点即播"，换"必然流畅"。**与 AE RAM Preview 完全同款心智**。

**(B) 多进程填充（S2'）** —— 边播边追、绕 GIL、工作量大：

- N 个 offscreen 进程并行渲未来帧 → 主进程收进缓存（复用导出多进程基建：`_render_worker_init` 等）。
- 把填充吞吐抬过实时 → look-ahead 缓冲不再被抽干 → **首播也能边播边追到满帧**。
- 工作量 **L–XL**；风险高：IPC（每帧 ~13MB，需共享内存 / `QSharedMemory` / mmap）、进程/字体/缓存复制、
  seek 时取消在途任务、Windows spawn 开销。
- **先做可行性探针**（见 §6）再决定。

## 4. 这条路解决什么 / 不解决什么（对齐主文档 §10.3 两轴）

| 诉求 | P-A 帧缓存 | P-B look-ahead | P-C(A) AE预渲 | P-C(B) 多进程 |
|---|---|---|---|---|
| 重播/seek/scrub 已渲段流畅 | ✅ 立即 | ✅ | ✅ | ✅ |
| 含轻松段混合工程平滑掉帧突发 | ⚠️ 部分 | ✅ | ✅ | ✅ |
| **首次连续播放纯重 utopia 段满帧** | ❌ | ❌（缓冲抽干）| ✅（需等待）| ✅（边播边追）|
| 同步正确性（音画不漂）| 正交（见主文档 §10.9 已做的音频锚定 + 单播放器）|

> **核心诚实点**：帧缓存**不会让单帧渲染变快**。它把成本**时移**（提前/空闲算）并**复用**（存帧）。
> 纯重 utopia 的首播，要么**等**（AE 式），要么**真并行填充**（多进程）。这两条之外没有第三种物理可能。

## 5. 与既有代码 / 开关的关系

- **改造点集中在 `preview_async.py` + `preview_graphics.py`**：worker 加 cache 写入与 look-ahead 投递；
  GUI `set_time`/`request` 加命中查询。`painter.py` 渲染核心**不动**。
- **复用现有失效范式**：`set_state` / `set_render_target` → 清缓存（同 `LayerCache.clear`）。
- **复用导出基建**：P-C(B) 多进程直接借 `engine/renderer.py` 的 `_render_worker_init` / chunk 切分 /
  offscreen worker 池（字体一致性已解决）。
- **复用条带逻辑**：缓存降本可借 `KROK_SUBTITLE_RENDER_STRIP` 的纵向并集 band。
- **新增开关**（沿用 `KROK_SUBTITLE_*` + 默认关、A/B 验证后转开 的项目范式）：
  `KROK_SUBTITLE_PREVIEW_FRAME_CACHE`（P-A）、`KROK_SUBTITLE_PREVIEW_LOOKAHEAD`（P-B）、
  `KROK_SUBTITLE_PREVIEW_MP_FILL`（P-C(B)）。
- **不影响导出**：本方案只动**预览**；导出本就离线逐帧 + 已多进程（`KROK_SUBTITLE_RENDER_WORKERS`）。

## 6. 落地步骤与验收

1. **基线测量**（先量再做）：`scripts/profile_preview.py --utopia --glow` 记录当前——首播 fps、
   重播 fps、seek 后命中率（现为 0）、worker 平均填充率（帧/s）。
2. **P-A**：帧缓存环 + 命中即取。验收：重播同段 fps、seek 回已渲处即满帧；纯像素回归（命中帧 == 重渲帧）。
3. **P-B**：look-ahead 预填。验收：混合工程（含轻松段）掉帧突发下降；纯重 utopia 段不劣化（退回 P-A）。
4. **P-C 决策前置探针**（关键，决定走 A 还是 B）：
   - 量 worker **单线程平均填充率 vs 实时**——若 < 实时，P-B 对纯 utopia 必抽干，确认是否需要 P-C。
   - P-C(B) 前必做 **IPC 吞吐探针**（参照主文档 §10.6 `bench_parallel_paint.py` 的 multiprocess 变体）：
     量 N 进程经共享内存填充 utopia+glow 帧的净吞吐（扣除 IPC/spawn），确认能否 ≥ 实时 × 安全系数。
5. **统一验收**：重特效工程播放中卡顿率显著下降、seek 趋零重渲；内存占用在环形上界内可控。

## 7. 待产品 / 接手者确认

1. **"即点即播" vs "等预渲必满帧"**：P-C(A) AE 式要求用户先等一段预渲（像 AE 按空格后等 RAM Preview）。
   产品能否接受重段的"渲染中…"等待？若**不可接受 → 只能走 P-C(B) 多进程**（成本高得多）。
2. **缓存内存上界**：环形窗口取多大（±1s ≈ 780MB @device 全屏；条带化可降至 ~1/3）?
   是否接受缓存降本带来的画质取舍（条带 / logical 分辨率 / 压缩）?
3. **目标场景画像**：实际工程是"全程重 utopia"还是"utopia 点缀 + 大量普通行"?
   前者 P-B 收益有限、几乎必须 P-C；后者 P-A+P-B 可能已够。**建议先拿真实工程量 §6.1 基线再定档**。

## 8. 这个缓存能加速导出吗?（诚实评估）

> 追加评估（2026-06-24）：问「§3 的帧缓存是否也能加速导出」。**直接结论：§3 那种「按帧索引的输出帧
> 缓存」(P-A) 基本不能加速导出**——但有一个**不同机制**的缓存能跨过去，值得单列。

### 8.1 为什么 P-A 帧缓存对导出无效

导出（[`engine/renderer.py`](../krok_helper/subtitle_render/engine/renderer.py)）的工作模式与预览**根本不同**：

- **每个 `frame_index` 只渲一次、严格按时间顺序、渲完即喂 ffmpeg pipe**
  （`_write_frames_single` :611 / `_frame_bytes` :592：有内容就画进复用 buffer → `_image_bytes` → 写 stdin）。
- **没有重播、没有 seek、没有逐帧 scrub**——而这三件事正是 P-A 缓存唯一能加速的（命中即取）。
  导出里每帧渲一次后再不回访 → **缓存没有任何「命中」机会，纯属空转 + 占内存**。
- 导出**不受墙钟约束**：它本就以最快速度连渲（不是实时回放）→ **look-ahead(P-B) 概念对它无意义**
  （它一直在"全速向前渲"）。
- 导出**已经并行**：多进程 worker（`KROK_SUBTITLE_RENDER_WORKERS`，默认 = CPU 核数）+ 条带
  （`KROK_SUBTITLE_RENDER_STRIP`）+ **空帧短路**（`frame_has_content` 为假 → 写预存全透明字节）。

> 一句话：P-A 缓存解决的是「**同一帧被多次需要**」；导出每帧只需要一次，**无可缓存性**。

### 8.2 真正跨预览/导出的，是另一类缓存——而它已经在用

需要区分**两种"缓存"**：

| | 缓存对象 | 复用维度 | 帮预览? | 帮导出? | 现状 |
|---|---|---|---|---|---|
| **输出帧缓存**（本文 P-A）| 整帧字幕 QImage，键 = `frame_index` | **跨次播放/seek**（同一帧多次取）| ✅ | ❌（每帧只渲一次）| 本方案待做 |
| **bake / 图层缓存** | 单个静态元素位图（行/glyph run/glow），键 = 与 t 无关的静态签名 | **跨帧**（同一行 bake 一次，可见的每帧都复用）| ✅ | ✅ | **已实现且默认开**（`LayerCache`/`_RUN_GLOW_CACHE`/before-layer/glow）|

→ **真正同时加速两条管线的缓存（bake 层）早已落地**（主文档 §1.3 / P0），这才是导出 P0 提速的来源。
**本文的输出帧缓存与它正交，不是同一回事。**

### 8.3 那导出还有没有「缓存类」提速空间?——有一个，且对两条管线都管用

**(a) 相同帧变更检测短路（change-detection / `vo_change_id` 思路，主文档 §10.6 列过）** —— 推荐评估：

- 现有空帧短路只处理「整帧无内容」。可**推广到「非空但与上一帧逐像素相同」**的帧：
  若 `frame_index=N` 的**字幕内容签名 == N−1**（无动画、无扫光进度变化、无信号闪烁），
  **直接复用上一帧已渲字节**，跳过 `_paint_overlay_strip` + `_image_bytes`。
- **命中场景**：整词唱完后**静止保持**的段（停在画面上不动）、句间停顿但字幕仍在的段 → 导出里这类帧
  **逐字节相同**，现在却被逐帧重渲重编码。**对导出和预览都省**。
- **不命中场景**：**utopia 全程抖动、扫光推进中** → 每帧签名都变 → 无收益（与 P-A 对 utopia 的局限同源）。
- 代价/风险：**低**。只需一个「内容签名」函数（量子化扫光进度 + 动画状态 + 信号状态）+ 上帧字节缓存 1 张。
  与多进程兼容（每 worker 在自己 chunk 内做帧间比较；跨 chunk 边界放弃复用即可）。
- 验收：静止保持段导出帧/s 上升；纯像素回归（复用帧 == 重渲帧）；utopia 段不劣化。

**(b) 磁盘预览帧复用（PR「Use Previews on export」思路）** —— 多数情况**不适用**，仅作记录：

- 设想：预览已渲好的帧落盘，导出直接复用，省导出重渲（PR 的"导出时使用预览文件"）。
- **致命错配**：预览按 **device/预览分辨率**渲（常更低、随 DPR 变、可能是条带），导出固定 **1920×1080 全幅**。
  分辨率/范围不一致 → **复用即降质**（与 PR「预览分辨率 < 导出分辨率时画质下降」同理）。
- 仅当"预览就按导出全分辨率渲"才可复用，但那等于放弃"低分辨率预览换流畅"（约束 3★）→ **得不偿失**。
- 结论：**不推荐**，除非未来出现"所见即所得全分辨率预览"需求。

### 8.4 小结

| 提速手段 | 帮预览 | 帮导出 | 建议 |
|---|---|---|---|
| 输出帧缓存（P-A/B/C）| ✅ | ❌ | 预览专用，本文主体 |
| bake/图层缓存 | ✅ | ✅ | **已默认开**（不用再做）|
| 相同帧变更检测短路 (8.3a) | ✅ | ✅ | **建议评估**：低风险，静止保持段两端都省；utopia 无效 |
| 磁盘预览帧复用 (8.3b) | — | ⚠️ | 不推荐（分辨率错配 / 与低分辨率预览冲突）|
| 多进程（导出已用 / 预览 P-C(B)）| ✅(待做) | ✅(已用) | 真并行，绕 GIL |

> **回答原问题**：本文这套帧缓存**不能**加速导出（导出无重播/seek、每帧只渲一次）。
> 真正两条管线通吃的是**已落地的 bake 层缓存**；导出端还能再榨的"缓存类"提速是
> **相同帧变更检测短路 (8.3a)**——低风险、对静止保持段有效、对 utopia 无效，建议另行评估落地。
